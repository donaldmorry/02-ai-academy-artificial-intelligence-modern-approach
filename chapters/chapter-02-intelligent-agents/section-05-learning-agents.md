## Why Learning Agents?

The architectures in [Section 2.4](./section-04-structure-of-agents.md) - reflex, model-based, goal-based, utility-based - assume a designer supplies **rules, models, goals, or utilities**. That works until:

- The environment is **too complex** to model by hand (vision, natural language)
- Dynamics **change** (spam tactics evolve, traffic patterns shift)
- The designer **does not know** the right behavior in advance

A **learning agent** improves performance on a [performance measure](../../GLOSSARY.md#performance-measure) from **experience** generated by its own actions. This is Russell and Norvig's bridge to [Course 1](https://github.com/donaldmorry/01-ai-academy-applied-ml-engineering/blob/main/chapters/chapter-01-machine-learning/section-01-what-is-machine-learning.md) and Chapters 19-22.

> **Readable form:** A learning agent is a regular agent plus a homework grader and a study coach - it tries things, gets feedback, and updates how it decides.

Humorous analogy: a non-learning agent is a **printed cookbook**. A learning agent is a cook who burns toast twice, adjusts the timer, and eventually wins a baking show - provided the critic actually tastes the output.

## The Four Components

Russell and Norvig decompose learning agents into four conceptual parts:

```
                    ┌─────────────────┐
   percepts ───────►│  PERFORMANCE    │──────► actions
                    │    ELEMENT    │
                    └────────┬────────┘
                             │ feedback (percepts, rewards)
                             ▼
                    ┌─────────────────┐
                    │     CRITIC      │──► learning signal
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼                             ▼
     ┌─────────────────┐          ┌─────────────────┐
     │    LEARNING     │          │    PROBLEM      │
     │    ELEMENT      │          │   GENERATOR     │
     └─────────────────┘          └─────────────────┘
              │                             │
              └──────────► updates performance element
```

### 1. Performance Element

The part responsible for **selecting external actions** - the agent you would deploy today. It may be any architecture from [Section 2.4](./section-04-structure-of-agents.md): reflex rules, a planner, a neural network policy.

Think of it as the **employee on the factory floor**. Everything else is HR and training.

### 2. Learning Element

Uses feedback from the critic to **modify** the performance element:

- Update transition model parameters
- Refine utility estimates
- Retrain a classifier
- Adjust rule weights

From Mitchell's definition ([Section 1.1](../chapter-01-introduction/section-01-what-is-ai.md)):

> A program **learns** from experience $E$ with respect to task $T$ and performance measure $P$, if its performance at $T$, as measured by $P$, improves with experience $E$.

| Component | Chess learning agent | Spam filter learning agent |
|-----------|---------------------|---------------------------|
| **$T$** | Win games | Classify mail correctly |
| **$E$** | Self-play, human games | Labeled inbox, user corrections |
| **$P$** | Win rate, Elo rating | Precision/recall on held-out mail |

### 3. Critic

Tells the learning element **how well the agent is doing** on the [performance measure](../../GLOSSARY.md#performance-measure). The critic observes the world (or percepts) and returns a **reward signal** or **training label**.

```python
def critic(percepts, environment_state, performance_measure):
    # Simplified: scalar feedback for learning
    return performance_measure.score(environment_state, percepts)
```

> **Readable form:** The critic is the scoreboard - it watches outcomes and tells the learner whether things got better or worse.

**Important:** the critic uses **true outcomes** when available (external performance measure), not just the agent's beliefs. A fraud detector's critic might use **confirmed chargebacks** weeks later - delayed feedback.

### 4. Problem Generator

Suggests **exploratory actions** - experiments that may uncover better policies at the cost of short-term performance. Without it, the agent **exploits** known good behavior forever and never discovers improvements.

```python
def problem_generator(state, known_actions):
    # Occasionally propose informative or novel actions
    if should_explore(state):
        return random_untried_action(state)
    return None  # performance element decides otherwise
```

> **Readable form:** The problem generator is the curious friend who says "try the weird menu item" so you learn something new.

Exploration vs. exploitation is central to reinforcement learning (Chapter 22) and bandits (Chapter 17).

## Learning Agent Control Loop

```python
def learning_agent_loop(environment, max_steps=1000):
    percepts = []
    state = initial_internal_state()

    for t in range(max_steps):
        percept = environment.get_percept()
        percepts.append(percept)

        # Problem generator may suggest exploration
        exploratory = problem_generator(state, legal_actions(state))

        if exploratory is not None:
            action = exploratory
        else:
            action = performance_element(percepts, state)

        environment.execute(action)
        state = update_state(state, percept, action)

        # Critic evaluates after action effects observed
        feedback = critic(percepts, environment.true_state(), PERFORMANCE)

        # Learning element updates performance element
        learning_element.update(
            percepts=percepts,
            action=action,
            feedback=feedback,
            state=state,
        )

        if environment.is_terminal():
            break
```

> **Readable form:** Sense, maybe experiment, act, get graded, study, repeat.

Compare to [Course 1](https://github.com/donaldmorry/01-ai-academy-applied-ml-engineering/blob/main/chapters/chapter-01-machine-learning/section-lab-01-customer-segmentation-and-flower-classification.md) train/test loops: learning happens **online** inside the agent loop, not only in an offline notebook.

## What Gets Learned?

Depending on architecture, the learning element updates different internals:

| Base architecture | Learnable components | Course 2 chapters |
|-------------------|------------------------|------------------|
| Reflex | Rule weights, feature detectors | 19 |
| Model-based | Transition/sensor models | 12, 20 |
| Goal-based | Heuristics, domain models | 19, 21 |
| Utility-based | Utility function parameters | 16, 22 |

**Representation learning** (Course 3) discovers features automatically - the learning element changes not just weights but **what the agent perceives**.

## Learning and Autonomy

[Section 2.2](./section-02-good-behavior-and-rationality.md) introduced **autonomy**: reliance on own experience vs. designer knowledge.

| Stage | Autonomy | Description |
|-------|----------|-------------|
| Cold start | Low | Designer rules only |
| After data | Medium | Learned models augment rules |
| Mature | High | Agent adapts to local conditions |

A [rational agent](../../GLOSSARY.md#rational-agent) with a learning element can become **more rational over time** - if the critic aligns with true goals. Misaligned critics teach the wrong sections ([Section 1.5](../chapter-01-introduction/section-05-risks-and-benefits-of-ai.md)).

## Batch vs. Online Learning

| Mode | Experience | Example |
|------|------------|---------|
| **Batch** | Fixed dataset, multiple passes | [OLS regression](https://github.com/donaldmorry/01-ai-academy-applied-ml-engineering/blob/main/chapters/chapter-02-regression-models/section-02-linear-regression.md) on CSV |
| **Online** | Stream of percepts, incremental updates | Spam filter from user clicks |
| **Active** | Agent chooses informative queries | Problem generator + labeling |

Learning agents in dynamic [PEAS](../../GLOSSARY.md#peas) environments ([Section 2.3](./section-03-the-nature-of-environments.md)) favor **online** and **active** modes.

## Critic Design Challenges

### Delayed rewards

Loan default appears months after approval. Credit assignment: which decisions caused the outcome? Chapter 22 addresses temporal difference learning.

### Sparse rewards

Robot reaches goal rarely during early training. **Reward shaping** and **curriculum learning** help - but can bias behavior if shaped rewards diverge from true $P$.

### Proxy metrics

Optimizing click-through rate as critic proxy for user satisfaction invites manipulation. Same rationality section: **critic must approximate true performance measure**.

## Problem Generator in Practice

Exploration strategies:

| Strategy | Behavior | Risk |
|----------|----------|------|
| $\epsilon$-greedy | Random action with probability $\epsilon$ | Wastes steps |
| Optimism under uncertainty | Try actions with high uncertainty | Can be aggressive |
| Information gain | Choose action that reduces model uncertainty | Computationally costly |

In [supervised learning](https://github.com/donaldmorry/01-ai-academy-applied-ml-engineering/blob/main/chapters/chapter-01-machine-learning/section-03-supervised-vs-unsupervised-learning.md), the "problem generator" is passive - data is given. In RL, **data is created by acting** - a profound shift.

## Worked Example: Adaptive Thermostat

| Component | Implementation |
|-----------|----------------|
| **Performance element** | Rule: heat if temp below setpoint minus margin |
| **Critic** | Occupant comfort surveys + energy bill |
| **Learning element** | Adjust setpoint schedule and margins |
| **Problem generator** | Try slightly lower night temperatures occasionally |

Over weeks, the agent learns you tolerate 67°F at night but want 70°F by 6 AM - without programmer hard-coding your habits.

## Failure Modes

**Critic hacking:** agent maximizes proxy reward while violating true goals (simulation exploits).

**Catastrophic forgetting:** new data overwrites old skills (fine-tuning LLMs without replay).

**Exploration disasters:** problem generator tries unsafe actions in physical robots - need constraints and safe exploration.
